    Q: Can't run device with bus clock above 1 MHz?
    A: Make sure device runs in normal (RUN) mode. Check your code as well as defines in the appconfig.h for potential entry to very low power run (VLPR) mode.


    Q: How to create specific on-chip peripheral configuration?
    A: Select the configuration structure with the closest functionality from peripheral's header file and copy and place it to the appconfig.h file of your application. After renaming the structure apply SET and CLR macros for each register field to initialize on-chip peripheral behavior according to your needs. Use SET macro whenever register bit or bit fields shall be set to specified value. Use CLR macro whenever to clear register bit or bit fields. Such created specific on-chip peripheral structure can be passed as an argument to the peripheral init function. Refer to Creating Application Specific Peripheral Configuration Structure for more details.


    Q: Can't debug IRTC driver?
    A: A protection mechanism is built in to the IRTC module to protect its registers against spurious writes by any run-away code. The protection mechanism requires the CPU to unlock IRTC registers for writing. After unlocking the registers, the CPU has a window of 2 seconds for updating the register space. Any updates beyond the unlock window would terminate by Hard Fault.


    Q: Software freezes in default_isr function?
    A: When the device receives a fault or an on-chip peripheral interrupt event which service routine is not installed then the core starts executing default_isr() function. The default_isr() function enters the core to an infinite loop, preserving the system state for examination by a debugger. Note that if a watchdog timer has been enabled then the device resets when watchdog timer refreshes period elapses.
    void default_isr(void)
    {
    for (;;)
    {
    }
    }


    Q: How to register handler for HardFault exception?
    A: Program an application specific hardfault_isr() function in your main.c file.
    void hardfault_isr(void)
    {
    for (;;)
    {
    }
    }
    Register new handler function in the appconfig.h file.
    extern void hardfault_isr (void);
    #define VECTOR_003 hardfault_isr


    Q: Is it possible to accelerate on-chip peripheral programming via Bit Manipulation Engine (BME)?
    A: By default, software drivers use BME read-modify-write capability for programming all on-chip peripherals. The BME support can be disabled by inserting #define EMULATE_BME_IO_ACCESS into appconfig.h file.


    Q: Can I use Micro-Trace Buffer tracing capabilities?
    A: Yes, you can use Micro-Trace Buffer (MTB) debugging capabilities; to enable MTB insert #define MTB_RAM_RESERVE into appconfig.h file. This define reserves trace buffer in RAM for use by the MTB. The trace buffer starts at address 0x1FFFF000 and it has size of 64-bytes. You can also modify size of the trace buffer for example to 256-bytes by the following definition: #define MTB_RAM_SIZE 0x100. Note that other settings of the development tool might be necessary to activate MTB - for more details refer to user's manual of the respective tool .


    Q: Can I define a variable that is not initialized by the startup function?
    A: Yes, linker files support placing non-initialized variables. For example, the volatile uint16 noninit_var variable will be non-initialized by the startup function if defined as follows:
    __no_init volatile uint16 noninit_var; /* IAR EWARM 6.5x */
    volatile uint16 noninit_var __attribute__ ((section(".noinit"))); /* CW MCU 10.x, CrossWorks 2.3.3 */



    Q: Can I define a function that executes from RAM?
    A: Yes, use the following templates to define void foo(void) function into RAM:
    /* IAR EWARM 6.5x */
    __ramfunc void foo(void); /* prototype */
    __ramfunc void foo(void) { } /* definition */
    /* CW MCU 10.x, CrossWorks 2.3.3 */
    void foo(void) __attribute__ ((section(".fast"))); /* prototype */
    void foo(void) { } /* definition */
