I/O Benchmark
Basic I/O Operations

The following benchmark application demonstrates capabilities of the Bit Manipulation Engine (BME) and IOPORT, a special single-cycle interface with the ARM Cortex-M0+ core, for GPIO pin control. The code sets and clears PTF3 and PTF5 using BME and IOPORT, respectively. Figure 1 shows timings of the GPIO pin toggling for non-optimized code - green waveform represents pin PTF3 and yellow waveform pin PTF5.

Figure 1. PTF3 and PTF5 toggling - no optimization.

Figure 2 shows timings of the GPIO pin toggling for code optimized for speed - green waveform represents pin PTF3 and yellow waveform pin PTF5.

Figure 2. PTF3 and PTF5 toggling - optimized for speed.

Timings and code sizes in a while(1){..} loop for non-optimized and optimized codes are summarized below:

Benchmark (optimization)BME IOPORT
Execution time (none) 4.352us 4.352us
Execution time (speed) 2.367us 1.536us
Code size (none) 16 bytes 28 bytes
Code size (speed) 4 bytes 12 bytes
Benchmark application:
/******************************************************************************
* (c) Copyright 2010-2013, Freescale Semiconductor Inc.
* ALL RIGHTS RESERVED.
******************************************************************************
* bme_test.c
******************************************************************************/
#include "drivers.h"
void main (void)
{
SIM_EnableModule(PORTF);
PORT_Init (PORTF, PORT_MODULE_LED_MODE, PIN3|PIN5);
GPIO_Init (GPIOF, GPIO_OUT_LOGIC0_MODE, PIN3|PIN5);
while(1)
{
// Pin toggled via Bit Manipulation Engine (BME)
GPIO_Set (GPIOF , PIN3);
GPIO_Clr (GPIOF , PIN3);
// Pin toggled via IO PORT
GPIO_Set (FGPIOF, PIN5);
GPIO_Clr (FGPIOF, PIN5);
}
}
Summary:
The GPIO control benchmark application demonstrated that accesses accelerated via Bit Manipulation Engine (BME) results in a smaller code in comparison to the IOPORT accesses. This is natural since most BME commands; a single core read or writes bus cycle is converted into an atomic read-modify-write, that is, an indivisible "read followed by a write" bus sequence. On the other hand, the IOPORT accesses or even software emulated BME commands result in a faster code execution but larger memory footprint.
Note
The BME accelerated accesses to on-chip peripherals are enabled by default. Insert #define EMULATE_BME_IO_ACCESS into appconfig.h file to bypass BME.
